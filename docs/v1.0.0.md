# clickup-axi v1.0.0 - scope

**Goal:** an agent can run a realistic ClickUp task workflow end to end -
not just read and set status. Complete the write loop (create + full edit),
add the discovery needed to target it, and keep the inline-hint ergonomics
uniform across every new field.

**Current (v0.6.0 tagged; later work unreleased on main, audited
2026-07-12):** read is strong - `tasks` (open tasks,
`--assignee`/`--space`/`--page`/`--fields`), `tasks <id>` (view +
comments + tags + parent/direct subtasks), `search` (rich filters,
relative date bounds).
`tasks edit` covers the full field set (status, assignees, priority,
name, due, body, tags, parent) and `tasks comment` works. `tasks
create` completes the write loop. `setup` installs the session-start
hook; structure discovery (`spaces`/`lists`) has landed. `tasks close`
(guarded), the config file with `default_list`, `tasks move`, and the
parent/subtask detail view all shipped 2026-07-12 (unreleased). The
planned v1 command surface is complete.

**Ordering principle:** steps are sorted by increasing entanglement. Early
steps reuse resolvers that already exist; `create` comes last among the
must-haves because it is the only step that forces the config file and
depends on list discovery. Each step ships on its own.

## Step 1: `tasks --assignee <name>` - close the read gap (DONE)

Shipped 2026-07-09 in v0.4.0. The tasks *list* command got the same
`--assignee` flag as `search`, so "show me Ting's open tasks" works.
The optional `--space` rider shipped alongside it.

- Default stays `me`; names resolve via `Team.ResolveMember`
  (exact -> unique substring -> ambiguous/none with inline candidates).
- Pure reuse of the search resolver - smallest possible step.
- Optional rider: `tasks --space <name|id>` (scope the listing the way
  `search --space` already does). Backed by the existing `ResolveSpace`,
  so it has no dependency on discovery (step 5) - same reuse pattern as
  `--assignee`. Space is the ClickUp term agents will reach for.

## Step 2: `tasks edit --assignee` - assignee mutation (DONE)

Shipped 2026-07-09 in v0.4.0 on branch `feature/edit-assignee`.

```
clickup-axi tasks edit <id> --assignee <who>   add an assignee
clickup-axi tasks edit <id> --unassign <who>   remove an assignee
```

- Additive add + explicit `--unassign` (chosen over replace-semantics
  and a `+name`/`-name` sign syntax). Both flags repeat and accept a
  comma-separated list; `<who>` is `me`, a member name, or an id.
- Composes with `--status` in one call.
- Idempotent no-ops (already-assigned add / absent-member unassign
  exit 0 with a stated "no changes"); same person in both flags is a
  usage error.
- New `clickup.UpdateTask(id, TaskEdit)` -> one `PUT /task/{id}` carrying
  `status` and `assignees.add/rem` together (status + assignee commit
  atomically); reuses `ResolveMember` via `resolveEditAssignee`.
- No-match suggests members inline (same recovery as search).

## Step 3: `tasks edit` - full field set (extends today's `--status`) (DONE)

Shipped 2026-07-10 in v0.5.0 on branch `feature/extend-edit`, adding `--priority`,
`--name`, `--due`, `--body`/`--append-body`, and `--add-tag`/`--remove-tag`.
Tags are validated against the space's tag list up front; the write
applies tags first and the atomic `PUT` last, rolling the tag ops back
on a later failure so the edit stays all-or-nothing. `tasks <id>` now
also renders tags.

```
clickup-axi tasks edit <id> [--status --assignee --priority --body --name --due --add-tag --remove-tag]
  --priority <p>       urgent | high | normal | low | none
  --body "<markdown>"  replace description
  --name "<title>"     rename
  --due <date>         set/clear due date
```

- Accept multiple fields in one call.
- Builds the human -> ClickUp mappings (priority 1-4, due epoch ms,
  markdown body) that `create` reuses in step 6.

### Validation architecture (already in place from step 2, extend per field)

The edit path is **validate-all-then-write**, not write-then-explain, and
it stays that way as fields are added. Every field is checked in one
pre-flight pass before the single atomic `UpdateTask` PUT. This is the
deliberate answer to "one bad field in a four-field edit" - the atomic
PUT means an invalid field would otherwise block the whole edit *after*
a wasted round trip, so instead:

- **Collect, don't bail.** Each field appends to a `fieldErrs` slice
  instead of returning on the first failure, so a bad `--priority` and a
  bad `--status` in the same call are reported *together*. The agent
  fixes everything and retries once, rather than discovering errors one
  round trip at a time. `renderFieldErrors` prints them as a bulleted
  list (`N fields cannot be applied (nothing was changed)`).
- **Atomic, never partial.** All-or-nothing is intended: a partial apply
  would leave the task half-edited and force the agent to reconcile which
  fields landed. Since these are idempotent set-fields, re-running the
  corrected full command is cheap and safe - no partial-state bookkeeping.
- **Where each field validates:**
  - `--priority`, `--due` - purely local (enum / date parse), no API call.
  - `--status` - against the list's statuses (`listStatuses`, one GET;
    already wired). A status the list allows but the workflow forbids
    from the current state is only knowable at the API, so the atomic PUT
    stays the final gate and returns the raw translated error.
  - `--assignee` / `--unassign` - `resolveEditAssignee` against workspace
    members (already wired).
  - `--body`, `--name` - nothing to validate.
  - `--add-tag` / `--remove-tag` - against the list/space tags if cheap,
    else let the API be the gate.
- Each new field registers its check in the pre-flight pass and its
  mapping in the `TaskEdit` build; the write path does not change.

This removed the earlier asymmetry where a bad assignee (validated
locally) masked a bad status (validated only by the API).

## Step 4: `setup` session hook (DONE)

Shipped 2026-07-10 on branch `feature/setup-session-hook` (unreleased,
next tag). Ambient context per AXI §7, split into two commands:

- `setup` installs the session-start hook for Claude Code, Codex, and
  OpenCode. `--global` targets every host whose config dir exists;
  `--project` writes into the current directory's `.claude`/`.codex`
  configs (OpenCode is global-only). Rerunning repairs a moved binary
  path and is otherwise a no-op; `--remove` uninstalls, touching only
  clickup-axi's own entries. Host config knowledge lives in
  `internal/hostcfg`.
- `context` is the hook's payload: the user's 5 most urgent open tasks
  (due-soonest first, total stated) behind a hard 5-second budget. It
  always exits 0 and degrades to a one-line reason (not authenticated,
  workspace unpinned, API unreachable) so a broken network can never
  break a session start.

## Step 5: structure discovery - `spaces` / `lists` (DONE)

Implemented 2026-07-10 on branch `feature/lists-and-spaces` (unreleased,
next tag).

Create and move are unusable if the agent must guess list ids. This step makes
the ClickUp hierarchy explicit before `tasks create` arrives in step 6, without
adding configuration or any mutation.

Implementation details and decisions: [Step 5 spaces and lists design](superpowers/specs/2026-07-10-spaces-lists-design.md).

```
clickup-axi spaces
clickup-axi lists --space <name|id> [--archived]
```

- `spaces` selects the workspace through the existing `SelectTeam` policy and
  lists its active spaces as `spaces[N]{id,name}`. Results sort
  case-insensitively by name, then id. A zero result is explicit; a non-empty
  result hints at `clickup-axi lists --space "<name|id>"`.
- `lists` always requires `--space`, so an agent explicitly selects the project
  whose lists it wants to inspect rather than causing a hidden workspace-wide
  scan. It reuses `ResolveSpace`, including numeric ids, exact/unique-substring
  name matching, and the existing inline ambiguity/no-match recovery.
- List discovery includes both folderless lists and lists in folders inside the
  selected space. It renders `lists[N]{id,name,folder}`; folderless lists use
  the explicit `(folderless)` marker so duplicate list names stay distinguishable.
- The default view contains active lists only. `--archived` is lists-only and
  returns archived lists only; archived-space discovery is intentionally out of
  scope.
- Missing `--space`, missing flag values, and unknown flags are usage errors
  (exit 2) with valid recovery commands inline. Empty list results are stated
  explicitly. Any required folder or folderless-list API failure is translated
  through the existing error path and returns no partial inventory.

### Implementation

- Add ClickUp adapter support for folderless lists and folders with their nested
  lists. Normalize results into a reusable list reference carrying id, name,
  and folder context, ready for step 6 list-name resolution.
- Add command dispatch, concise per-command help, and `internal/cli/surface.go`
  entries. Update README examples and `internal/cli/skill_template.md`, then
  regenerate `skills/clickup-axi/SKILL.md`.
- Keep a separate `folders` command, default-list configuration, and task
  creation out of this step.

### Tests and handoff

- Assert exact stdout and exit codes for populated/empty spaces, active and
  archived lists, folderless and folder-contained lists, deterministic order,
  flag validation, space-resolution failures, and translated upstream failures.
- Verify `go vet ./...`, `go test ./...`, and generated-skill freshness.
- Commit frequently in coherent stages: adapter; command/output tests; then
  command surface, README, and generated skill.

## Step 6: `tasks create` - the headline gap, done last for a reason (DONE)

Implemented 2026-07-12 on branch `feature/tasks-create` (unreleased,
next tag). Decisions taken: a list *name* requires `--space` (list
names are only unique within one space; no workspace-wide scan), a
numeric list id works alone; `--parent` derives the list from the
parent task and refuses a contradicting `--list`; `--list` stays
strictly required otherwise - the config file with `default_list` is
the immediate follow-up step. (Relative due dates were deferred at
create time but shipped right after - see "Shipped beyond the plan"
below.) Same validate-all-then-write pre-flight as
edit, then one atomic POST (tags ride in the create body, so no
rollback machinery). The confirmation echoes the server-stored id,
list, defaulted status, and url.

An agent literally cannot make work today - but create is the one step
that forces the config file and needs list discovery (step 5) to be usable.

```
clickup-axi tasks create "<name>" --list <name|id> [flags]
  --status "<status>"      initial status (else list default)
  --assignee <who>         me | member name | id  (repeatable)
  --priority <p>           urgent | high | normal | low  (none = unset)
  --body "<markdown>"      description (markdown)
  --due <date>             YYYY-MM-DD (accept "friday"/"+3d" if cheap)
  --parent <id>            create as a subtask
  --tag <t>                repeatable
```

- `--list` required (or a `CLICKUP_AXI_DEFAULT_LIST` env / `default_list`
  config for the common case - this is the genuine forcing need for the
  config file).
- Resolve list name -> id (reuse the step-5 resolver); refuse an ambiguous
  name and show the matches.
- Print the created id + url.
- ClickUp mapping: `POST /list/{id}/task`; priority name->1-4; due->epoch ms;
  `--body` -> `markdown_description` (all built in step 3).

## Shipped beyond the plan (unreleased, audited 2026-07-12)

Work that landed after step 6 without being scoped here:

- **Relative dates + workspace timezone** (`feature/relative-dates`):
  `--due` on create/edit and the `search` `--updated-after`/`--before`
  bounds accept signed offsets (`+3days`, `-1week`). Offsets resolve
  and dates render in the ClickUp workspace timezone (cached on disk
  for 24h), not the host's local zone.
- **`tasks edit --parent <id>`** (`feature/parent-tasks`): make a task
  a subtask or re-parent it, atomic with other fields in the same PUT.
  Same-list constraint enforced, ancestor chain walked to refuse
  cycles (`ParentWouldCycle`), clearing a parent reported as
  unsupported by ClickUp's API, already-parented is a stated no-op.
- **`--fields`** on `tasks`/`search`/task view (`feature/axi-audit-fixes`):
  additive columns (assignees, priority, tags, list, url) rendered
  from the response already fetched; unknown names aggregate into one
  exit-2 error.
- **`tasks --page N`**: listings beyond the first 100, with
  filter-preserving next-page hints.
- **AXI audit fixes**: task-view `url:` is now opt-in via `--fields
  url` (breaking); self-contained outputs drop static `help[]` hints
  (AXI section 9); locally decidable bad values exit 2 before any API
  call.

## Remaining work to v1.0.0 (roadmap, 2026-07-12)

Ordered by the same entanglement principle: smallest reuse first,
destructive last. Each step ships on its own.

1. **Step 7: config file with `default_list`** (DONE) - shipped
   2026-07-12 on branch `feature/config-settings` (unreleased).
   Decisions taken: flat-TOML files (personal
   `~/.config/clickup-axi/config.toml` next to the token, committable
   project `.clickup-axi.toml` at the git root, discovered by walking
   up); precedence explicit flag > `CLICKUP_AXI_DEFAULT_LIST` >
   project > personal; a `config` command (show with per-value
   provenance, `set` with API validation storing the resolved id,
   idempotent `unset`); and a `folder:<id|name>` value form that
   derives a sprint folder's current list at each create, so a
   biweekly sprint rollover needs no reconfiguration.
2. **Step 8: parent + subtasks in the `tasks <id>` view** (DONE) -
   shipped 2026-07-12 on branch `feature/parent-sub-on-view`
   (unreleased). The detail-only task fetch requests subtasks without
   bloating mutation lookups; the view renders immediate parent
   metadata and a `subtasks[N]{id,title,status}` table of direct
   children (or an explicit zero state). Parent metadata uses one
   internal-id follow-up so forced custom-id workspaces stay consistent.
3. **Step 9: `tasks move <id> --list <target>`** (DONE, shipped
   2026-07-12 on branch `feature/task-move`, unreleased). The API
   design check dissolved the original caveat: ClickUp has since
   shipped a dedicated v3 Move Task endpoint
   (`PUT /api/v3/workspaces/{ws}/tasks/{task}/home_list/{list}`), which
   changes only the home list in one call - no ClickApp dependency, no
   add/remove workaround. Live probes established the semantics the
   command is built on: the personal token works on v3 unchanged,
   statuses match across lists **by name**, and `status_mappings`
   (status-id pairs) are rejected as invalid when the target already
   has the status. Decisions taken: a separate `move` verb (a second
   endpoint cannot join edit's single atomic PUT); target resolution
   copies create's rule (name needs `--space`, numeric id alone);
   status handling refuses with the target's statuses inlined and
   `--status` picks the landing status explicitly - refused as
   superfluous when the target has the current status (never a hidden
   second write); already-home is a stated no-op; unguarded like `edit
   --status` because a move is fully reversible (the confirmation
   carries both list ids for a one-step move back). Custom-field
   transfer is deliberately not requested (out of 1.0 scope).
4. **Step 10: `tasks close <id>`** (DONE, shipped 2026-07-12 on branch
   `feature/close-tasks`, out of roadmap order - it depends on nothing
   from steps 7-9). Decisions taken: close = the list's **closed-type
   status** (not archive) - it matches ClickUp's own "Close" vocabulary,
   crosses exactly the visibility boundary the CLI already defines, and
   the agent never needs to know the status name; the obsolete-task
   cleanup need is deferred to an honestly named future `tasks archive`
   (post-1.0) reusing the same guardrail. The guardrail: without `--yes`
   the command is a dry run (exit 0 - the preview is its designed
   behavior) stating the exact status change; the binary never prompts
   (AXI), so the generated skill instructs the agent to relay the dry
   run and add `--yes` only after the user confirms. Already-closed is a
   stated no-op; the confirmation echoes the previous status as a
   one-step reopen hint. `tasks edit --status` deliberately stays
   unguarded: there the agent explicitly names a looked-up status.
5. **Releases** (updated 2026-07-12): **v0.6.0** shipped create,
   spaces/lists, setup, relative dates, `edit --parent`, `--fields`, and
   `--page`. With steps 7-10 now landed, the next cut is
   **v1.0.0-rc.1**. Roll the changelog, verify the generated skill is
   fresh, and tag only after the release gate passes.

After the tag: extend the bench with the newly possible tasks
(create-with-fields, reassign, set-due, multi-field edit, subtask) per
the benchmark payoff section.

## Cross-cutting (matters more than command count)

- **Inline hints on EVERY field** (the H2 thesis): invalid `--priority` echoes
  the valid set; `--assignee` no-match suggests members; `--list` no-match lists
  options; status already does this. Uniformity is the AXI edge, not tool count.
- **Ambiguity refusal:** a topic/name matching >1 task or list -> refuse and
  show candidates, never silently pick one.
- **Destructive guardrail:** `close` requires `--yes` (or print a
  what-would-happen dry-run first). An agent must not silently destroy state.
- **Human -> ClickUp mapping:** priority urgent|high|normal|low|none <-> 1-4/null;
  `--due` YYYY-MM-DD -> epoch ms; `--body` markdown -> `markdown_description`.
- **Output parity:** every command prints id + name (+ url), greppable, stable
  format; same "not found" / "ambiguous" phrasing as the read commands.
- **Exit codes:** 0 on success, non-zero on error, for scripting.

## Explicitly OUT of scope for 1.0 (defer to 1.x)

Dependencies (blocks / waiting-on), custom fields (workspace-specific, high
effort), time tracking, checklists, ClickUp Docs, attachments. MCP has these;
1.0 does not need parity. Ship the daily-driver task loop first.

`tasks delete` is deferred indefinitely - hard-destructive on an agent
path. `close` (terminal status) covers the finish-work need; clearing
out obsolete or duplicate tasks belongs to a future `tasks archive`
(post-1.0) behind the same guardrail.

## Benchmark payoff

Each of these unlocks bench tasks that cannot exist today because they are
MCP-only (the current 21 tasks test only the read/status/comment intersection):
create-with-fields, set-priority, set-due-date, reassign, move-between-lists,
create-subtask, multi-field edit. That is the "extend the bench" step, and it
makes the AXI-vs-MCP comparison broad and fair instead of only where they
already match.
