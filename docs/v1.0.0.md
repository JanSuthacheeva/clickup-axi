# clickup-axi v1.0.0 - scope

**Goal:** an agent can run a realistic ClickUp task workflow end to end -
not just read and set status. Complete the write loop (create + full edit),
add the discovery needed to target it, and keep the inline-hint ergonomics
uniform across every new field.

**Current (v0.3.0):** read is strong - `tasks` (open tasks, `--assignee`/
`--space`), `tasks <id>` (view + comments), `search` (rich filters). Mutation
is only `tasks edit --status` and `tasks comment --text`. No create; no other
editable fields.

**Ordering principle:** steps are sorted by increasing entanglement. Early
steps reuse resolvers that already exist; `create` comes last among the
must-haves because it is the only step that forces the config file and
depends on list discovery. Each step ships on its own.

---

## Step 1: `tasks --assignee <name>` - close the read gap (DONE)

Shipped 2026-07-09. The tasks *list* command got the same
`--assignee` flag as `search`, so "show me Ting's open tasks" works.
The optional `--space` rider shipped alongside it.

- Default stays `me`; names resolve via `Team.ResolveMember`
  (exact -> unique substring -> ambiguous/none with inline candidates).
- Pure reuse of the search resolver - smallest possible step.
- Optional rider: `tasks --space <name|id>` (scope the listing the way
  `search --space` already does). Backed by the existing `ResolveSpace`,
  so it has no dependency on discovery (step 5) - same reuse pattern as
  `--assignee`. Space is the ClickUp term agents will reach for.

## Step 2: `tasks edit --assignee` - assignee mutation (DONE)

Shipped 2026-07-09 on branch `feature/edit-assignee`.

```
clickup-axi tasks edit <id> --assignee <who>   add an assignee
clickup-axi tasks edit <id> --unassign <who>   remove an assignee
```

- Additive add + explicit `--unassign` (chosen over replace-semantics
  and a `+name`/`-name` sign syntax). Both flags repeat and accept a
  comma-separated list; `<who>` is `me`, a member name, or an id.
- Composes with `--status` in one call.
- Idempotent no-ops (already-assigned add / absent-member unassign
  exit 0 with a stated "no changes"); same person in both flags is a
  usage error.
- New `clickup.UpdateTask(id, TaskEdit)` -> one `PUT /task/{id}` carrying
  `status` and `assignees.add/rem` together (status + assignee commit
  atomically); reuses `ResolveMember` via `resolveEditAssignee`.
- No-match suggests members inline (same recovery as search).

## Step 3: `tasks edit` - full field set (extends today's `--status`) (DONE)

Shipped 2026-07-10 on branch `feature/extend-edit`, adding `--priority`,
`--name`, `--due`, `--body`/`--append-body`, and `--add-tag`/`--remove-tag`.
Tags are validated against the space's tag list up front; the write
applies tags first and the atomic `PUT` last, rolling the tag ops back
on a later failure so the edit stays all-or-nothing. `tasks <id>` now
also renders tags.

```
clickup-axi tasks edit <id> [--status --assignee --priority --body --name --due --add-tag --remove-tag]
  --priority <p>       urgent | high | normal | low | none
  --body "<markdown>"  replace description
  --name "<title>"     rename
  --due <date>         set/clear due date
```

- Accept multiple fields in one call.
- Builds the human -> ClickUp mappings (priority 1-4, due epoch ms,
  markdown body) that `create` reuses in step 6.

### Validation architecture (already in place from step 2, extend per field)

The edit path is **validate-all-then-write**, not write-then-explain, and
it stays that way as fields are added. Every field is checked in one
pre-flight pass before the single atomic `UpdateTask` PUT. This is the
deliberate answer to "one bad field in a four-field edit" - the atomic
PUT means an invalid field would otherwise block the whole edit *after*
a wasted round trip, so instead:

- **Collect, don't bail.** Each field appends to a `fieldErrs` slice
  instead of returning on the first failure, so a bad `--priority` and a
  bad `--status` in the same call are reported *together*. The agent
  fixes everything and retries once, rather than discovering errors one
  round trip at a time. `renderFieldErrors` prints them as a bulleted
  list (`N fields cannot be applied (nothing was changed)`).
- **Atomic, never partial.** All-or-nothing is intended: a partial apply
  would leave the task half-edited and force the agent to reconcile which
  fields landed. Since these are idempotent set-fields, re-running the
  corrected full command is cheap and safe - no partial-state bookkeeping.
- **Where each field validates:**
  - `--priority`, `--due` - purely local (enum / date parse), no API call.
  - `--status` - against the list's statuses (`listStatuses`, one GET;
    already wired). A status the list allows but the workflow forbids
    from the current state is only knowable at the API, so the atomic PUT
    stays the final gate and returns the raw translated error.
  - `--assignee` / `--unassign` - `resolveEditAssignee` against workspace
    members (already wired).
  - `--body`, `--name` - nothing to validate.
  - `--add-tag` / `--remove-tag` - against the list/space tags if cheap,
    else let the API be the gate.
- Each new field registers its check in the pre-flight pass and its
  mapping in the `TaskEdit` build; the write path does not change.

This removed the earlier asymmetry where a bad assignee (validated
locally) masked a bad status (validated only by the API).

## Step 5: structure discovery - `spaces` / `lists`

```
clickup-axi spaces                     # id + name of each space
clickup-axi lists [--space <name|id>]  # lists (id + name) to target
```

- Create/move are unusable if the agent must guess list ids. Search already
  resolves space/list names internally - expose it as listing commands.

## Step 6: `tasks create` - the headline gap, done last for a reason

An agent literally cannot make work today - but create is the one step
that forces the config file and needs list discovery (step 5) to be usable.

```
clickup-axi tasks create "<name>" --list <name|id> [flags]
  --status "<status>"      initial status (else list default)
  --assignee <who>         me | member name | id  (repeatable)
  --priority <p>           urgent | high | normal | low  (none = unset)
  --body "<markdown>"      description (markdown)
  --due <date>             YYYY-MM-DD (accept "friday"/"+3d" if cheap)
  --parent <id>            create as a subtask
  --tag <t>                repeatable
```

- `--list` required (or a `CLICKUP_AXI_DEFAULT_LIST` env / `default_list`
  config for the common case - this is the genuine forcing need for the
  config file).
- Resolve list name -> id (reuse the step-5 resolver); refuse an ambiguous
  name and show the matches.
- Print the created id + url.
- ClickUp mapping: `POST /list/{id}/task`; priority name->1-4; due->epoch ms;
  `--body` -> `markdown_description` (all built in step 3).

## Cross-cutting (matters more than command count)

- **Inline hints on EVERY field** (the H2 thesis): invalid `--priority` echoes
  the valid set; `--assignee` no-match suggests members; `--list` no-match lists
  options; status already does this. Uniformity is the AXI edge, not tool count.
- **Ambiguity refusal:** a topic/name matching >1 task or list -> refuse and
  show candidates, never silently pick one.
- **Destructive guardrail:** `close` requires `--yes` (or print a
  what-would-happen dry-run first). An agent must not silently destroy state.
- **Human -> ClickUp mapping:** priority urgent|high|normal|low|none <-> 1-4/null;
  `--due` YYYY-MM-DD -> epoch ms; `--body` markdown -> `markdown_description`.
- **Output parity:** every command prints id + name (+ url), greppable, stable
  format; same "not found" / "ambiguous" phrasing as the read commands.
- **Exit codes:** 0 on success, non-zero on error, for scripting.

---

## Explicitly OUT of scope for 1.0 (defer to 1.x)

Dependencies (blocks / waiting-on), custom fields (workspace-specific, high
effort), time tracking, checklists, ClickUp Docs, attachments. MCP has these;
1.0 does not need parity. Ship the daily-driver task loop first.

`tasks delete` is deferred indefinitely - hard-destructive on an agent path;
`close` (archive) covers the workflow need.

## Benchmark payoff

Each of these unlocks bench tasks that cannot exist today because they are
MCP-only (the current 21 tasks test only the read/status/comment intersection):
create-with-fields, set-priority, set-due-date, reassign, move-between-lists,
create-subtask, multi-field edit. That is the "extend the bench" step, and it
makes the AXI-vs-MCP comparison broad and fair instead of only where they
already match.
